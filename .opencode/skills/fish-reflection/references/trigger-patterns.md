# Trigger Patterns（触发模式识别）

fish-reflection只在以下三种模式下激活。本文档提供每种模式的具体识别信号、排除条件和边界案例。

## T1: 用户纠正/返工

### 识别信号

**直接纠正**（用户明确指出Agent产出有误）：
- "这不对"、"你搞错了"、"不是这样的"
- "这里应该是X，不是Y"
- "你忘了我说过的…"
- "我之前说了不要…"

**要求返工**（用户要求撤销或重做）：
- "重新做"、"改回去"、"再来一次"
- "撤销刚才的修改"
- "回到之前的版本"
- "这个方案不行，换一个"

**隐式纠正**（用户提供正确版本替代Agent产出）：
- 用户直接给出正确代码/文本，并要求替换Agent的输出
- 用户引用之前的约束条件，暗示Agent遗漏

### 排除信号（不激活）

- 用户改变需求："其实我想要的是…"（需求变更≠纠正错误）
- 用户追加要求："还需要加上…"（增量补充≠推翻返工）
- 用户探索方案："试试另一种方式？"（探索性迭代≠纠偏）
- 用户表达偏好："我更喜欢简洁的写法"（审美偏好≠对错判定）
- 用户修改自己之前的指令："算了，改成…"（用户自我修正≠Agent出错）

### 边界案例判定

| 场景 | 判定 | 理由 |
|------|------|------|
| 用户说"换个方案" | ⚠️ 需要判定 | 如果上一个方案执行失败→纠正；如果用户只是想探索→不激活 |
| 用户提供一段代码说"用这个" | ⚠️ 需要判定 | 如果替换Agent写的代码→纠正；如果作为新增输入→不激活 |
| 用户说"我说过不要动X文件" | ✅ 激活 | 明确指出Agent遗漏已知约束 |
| 用户说"格式改成表格" | ❌ 不激活 | 呈现形式偏好，非内容层面的错误 |

## T2: 重复失败（2+次连续）

### 识别信号

- 同一个命令（build/test/lint/deploy）连续2次返回非零exit code
- 同一个文件的同一类错误修了又出现（如lint error修复后新引入同类error）
- 同一个API调用连续超时或返回相同HTTP状态码（如连续两次422 Unprocessable Entity）
- 连续2次用不同修复策略解决同一问题但问题仍在（修复策略用尽）
- 连续2次delegation到subagent但subagent返回相同类型的失败

### 计数规则

- "同一操作"指目标和意图相同，不要求命令文本字符串完全一致
  - 例：`npm run build` 和 `npx tsc --noEmit` 都是"检查TypeScript编译"，算同一操作
- 中间穿插了其他不相关的操作，计数器归零重新开始
- 外部因素导致的不可用（DNS解析超时、CDN回源慢、GitHub API rate limit、SSH连接中断）不计入——参见AGENTS.md网络重试规则，先重试两次再考虑方案变更
- 用户主动要求"再试一次"不计入自动计数（用户已知悉并接受重试）

### 排除信号（不激活）

- 首次失败（只有1次，还不构成"重复"模式）
- 不同类型的错误碰巧在时间上连续出现（TypeError后面跟着NetworkError≠同一问题重复出现）
- 预期中的迭代试错过程（如用git bisect二分法定位引入回归的commit）
- 压力测试或负载测试中的预期失败（ab/wrk/locust产生的5xx属于被测量对象，非Agent行为失败）

### 失败分类矩阵

| 失败类型 | 示例 | 是否计入重复计数 |
|---------|------|--------------|
| 编译/构建错误 | tsc报类型不兼容、gcc报未定义引用 | ✅ 计入 |
| 测试断言失败 | pytest AssertionError、jest expect不匹配 | ✅ 计入 |
| 静态分析警告升级 | eslint error新增、mypy strict模式报错 | ✅ 计入 |
| 部署健康检查未通过 | HTTP 502/503/504、readiness probe超时 | ✅ 计入 |
| 网络瞬态中断 | ETIMEDOUT、ECONNRESET、DNS NXDOMAIN | ❌ 不计入 |
| 资源配额耗尽 | GitHub API 403 rate limit、Docker Hub pull limit | ❌ 不计入（先等待冷却再重试） |
| 权限不足 | EACCES、sudo required、SSH key rejected | ⚠️ 如果第一次已诊断出权限问题并修复但仍失败→计入 |

## T3: 显式请求

### 识别信号

**中文关键词**：
- "反思一下"、"复盘"、"总结经验"
- "为什么会出这个错"、"失败原因分析"
- "经验教训"、"教训是什么"
- "写成规则"、"固化经验"
- "沉淀一下"、"记录下来避免再犯"

**英文关键词**：
- "reflect"、"reflection"、"retrospective"
- "what went wrong"、"lessons learned"
- "postmortem"、"root cause analysis"
- "write this as a rule"、"capture this lesson"
- "blameless review"、"incident debrief"

### 级别选择启发式

| 用户请求特征 | 推荐级别 | 产出形式 |
|------------|---------|---------|
| 针对单个具体错误 | L1 | 内联3-5行 |
| 针对整个任务/会话的回顾 | L2 | `.opencode/reflections/YYYY-MM-DD-xxx.md` |
| 要求写成通用规则或团队指导 | L3 | `.opencode/reflections/guidance/xxx.md` |
| 用户说"简单说一下" | L1 | 即使问题复杂也压缩到L1 |
| 用户说"详细分析" | L2 | 即使问题简单也展开到L2 |
| 用户说"写成文档/规范" | L3 | 产出独立指导文件 |

### 与其他Skill的区分

| 用户说… | 路由到 | 理由 |
|---------|--------|------|
| "为什么这个bug出现" | 调试流程 | 焦点是修bug，不是反思行为模式 |
| "为什么我们老是犯这类bug" | fish-reflection L2/L3 | 焦点是行为模式，不是单个bug |
| "帮我review这段代码" | code review | 审查代码质量，非复盘 |
| "review一下我们这周的工作" | fish-reflection L2 | 回顾性复盘 |
| "这个项目哪里有问题" | QA流程 | 检查项目质量，非Agent行为复盘 |
| "我为什么总是写出有问题的代码" | fish-reflection L3 | 元认知层面的模式识别 |
