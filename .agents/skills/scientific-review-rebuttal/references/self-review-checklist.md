# Pre-Submission Self-Review Checklist for Scientific Papers

这份清单用于投稿前最后一轮自查，目标不是“打分好看”，而是提前发现最可能导致拒稿的问题。按照常见审稿关注点组织为六个维度：**novelty, technical soundness, evaluation, presentation, reproducibility, ethics**。每个维度都要给出证据，而不是只写“已检查”。

## 1) Novelty：贡献是否真实且可定位

审稿人首先问的不是“你做了多少工作”，而是“你到底新在哪里”。

### 核查项

1. 贡献是否在引言中明确列成可验证条目（而非口号）？
2. 每条贡献是否与现有工作有清晰差异边界？
3. 相关工作是否覆盖主要竞争路线，而非只选弱对手？
4. 是否把工程整合误写成理论创新？
5. 是否出现“贡献膨胀”：证据只支持局部，却声称普适领先？

### 红旗信号

- 贡献句子大量使用“novel”“first”“state-of-the-art”但无明确比较条件；
- related work漏掉该方向公认强基线；
- 结论措辞明显超出实验覆盖范围。

### 修复动作

- 把贡献重写为“在X条件下，对Y指标带来Z变化”；
- 在related work里补齐关键路线并明确关系；
- 收缩外推口径，显式写边界。

## 2) Technical Soundness：技术链条是否成立

技术正确性包含理论、算法、系统实现三层。任何一层有漏洞，审稿意见通常会集中在“soundness weak”。

### 核查项

1. 定义、符号、假设是否一致且无歧义？
2. 算法描述是否可执行（输入、输出、复杂度、关键步骤完整）？
3. 若有证明，是否存在未说明条件或跳步推导？
4. 实现是否与方法描述一致，是否有“论文写A，代码做B”的偏差？
5. 关键系统假设是否在真实环境可满足？

### 红旗信号

- “细节省略 due to space”覆盖了关键机制；
- 复杂度或安全假设未定义；
- 图示和文字描述不一致。

### 修复动作

- 补伪代码和必要引理；
- 在附录给出关键推导与边界条件；
- 对工程偏差写明影响及修正计划。

## 3) Evaluation：是否有足够证据支撑主张

评估是拒稿高发区。常见问题是实验多但不回答主张，或比较不公平。

### 核查项

1. 每条核心贡献是否映射到至少一个评估结果？
2. baseline是否足够强且设置公平？
3. 指标是否与主张一致（准确率主张不该只报吞吐）？
4. 是否做了必要ablation来隔离贡献？
5. 是否报告统计显著性与效应量，而非只给单次最好结果？
6. 是否披露失败案例、负结果、边界场景？

### 红旗信号

- 只比较弱基线；
- 只报平均值，不报方差和尾部指标；
- 结果图很多，但claim-evaluation map为空。

### 修复动作

- 补强基线与公平性说明；
- 增加针对性ablation；
- 把图表按“回答哪个问题”重组。

## 4) Presentation：表达是否清晰、结构是否可审

很多“misunderstanding”其实是写作结构问题，不是审稿人恶意。

### 核查项

1. 摘要是否包含具体结果（定量或明确发现），而非仅“we propose”？
2. 引言是否完成 problem-gap-insight-contribution 闭环？
3. 图表是否自解释（标题、单位、图例、关键结论）？
4. 章节衔接是否顺畅，术语是否前后一致？
5. 篇幅是否符合投稿要求，是否存在冗余叙述？

### 红旗信号

- 读完摘要仍不知核心结果；
- 图表引用错位或编号混乱；
- 同一概念有多个叫法。

### 修复动作

- 重写摘要第一段与最后一句；
- 统一术语表；
- 给每图一条“takeaway sentence”。

## 5) Reproducibility：第三方是否可复现主要结果

可复现不是“代码将开源”的承诺，而是“今天别人能按文档跑出接近结果”。

### 核查项

1. 是否提供代码版本、依赖、环境信息？
2. 数据来源、预处理与划分是否可追踪？
3. 关键参数、随机种子、运行脚本是否完整？
4. 图表是否可由原始结果脚本重建？
5. 若数据/代码受限，是否提供替代验证路径与限制说明？

### 红旗信号

- 参数散落在正文与附录，无法复现；
- 只给最终图，不给中间日志；
- 数据许可限制未说明。

### 修复动作

- 提供最小复现实验包；
- 补运行脚本与参数表；
- 对受限资源给出替代数据和偏差声明。

## 6) Ethics：伦理、双重用途与偏见风险

安全和AI研究常涉及双重用途风险，忽略伦理审查会直接触发重大负面评语。

### 核查项

1. 是否涉及人类受试者、隐私数据、敏感系统？是否需IRB或等价审批？
2. 方法是否可能被滥用于攻击、规避检测、歧视性决策？
3. 数据采集与标注是否存在群体偏差或地域偏差？
4. 是否在论文中披露风险与缓解措施？
5. 是否提供负责任发布策略（如限制细节、延迟公开、受控访问）？

### 红旗信号

- 论文只讲收益，不讲风险；
- 对敏感数据来源语焉不详；
- 把伦理问题完全推给“未来工作”。

### 修复动作

- 增加Ethics statement；
- 明确风险边界和缓解机制；
- 对可滥用细节做可解释删减并说明理由。

## 7) 评分与发布建议

可用简单分级辅助决策：

- A：可提交，仅需小修；
- B：基本可提交，需补若干关键说明；
- C：建议延期提交，评估或论证有明显缺口；
- D：不建议提交，核心主张与证据不匹配。

评分不是目的，关键是给出**可执行修订清单**：问题描述、影响范围、修复动作、责任人、截止时间。

## 8) 最终提交前的“硬门禁”

若以下任一项为“否”，建议不要提交：

1. 是否每条核心贡献都有对应评估结果？
2. 是否覆盖强基线并解释公平比较设置？
3. 是否披露至少两项真实局限？
4. 是否具备最小可复现实验说明？
5. 是否完成伦理风险披露（如适用）？

通过这五项不保证中稿，但能显著降低“可避免拒稿”。自查的价值在于：把审稿人会问的问题，提前在自己手里解决。
