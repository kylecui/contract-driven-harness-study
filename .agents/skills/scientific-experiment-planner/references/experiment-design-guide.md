# Experiment Design Guide for Systems and Security Research

实验设计不是“把方法跑起来然后报几个数字”，而是把研究主张转译成可检验命题，并用可重复的证据链回答“为什么你这个方法成立”。这份指南按固定顺序展开：**hypothesis → variables → controls → baselines → metrics → datasets/workloads → ablation → statistical tests → reproducibility**。顺序不是形式主义，而是为了防止研究中最常见的三类失败：主张不可检验、评估不对题、结果不可复现。

## 1) Hypothesis：从“贡献声明”到“可检验命题”

高质量假设具备三点：可证伪、可量化、有边界。比如在系统安全研究里，笼统地说“我们的方法更安全更高效”没有检验价值。应改写为：

- H1（有效性）：在相同攻击集上，方案A的检测召回率高于基线B至少5%。
- H2（性能）：在10GbE流量条件下，方案A的P99延迟增加不超过8%。
- H3（可部署性）：在给定CPU预算下，方案A吞吐不低于基线B的95%。

这种写法把“更好”分解为多个可测维度，并显式声明阈值。阈值可来自文献先验、业务约束或工程可接受区间，但必须说明来源。

## 2) Variables：变量建模决定结论边界

实验变量至少拆成三类：

- **Independent Variables（自变量）**：你主动改变的因素，例如检测窗口大小、策略刷新频率、模型蒸馏比例、XDP pipeline阶段数。
- **Dependent Variables（因变量）**：输出观测，如F1、误报率、吞吐、P95延迟、CPU占用。
- **Controls（控制变量）**：保持不变的条件，如硬件规格、内核版本、数据预处理方式、流量回放工具版本。

在安全研究中，攻击样本分布和“干净流量”比例经常被隐式改变，导致实验看似提升，实则不可比。把这类因素写入控制变量，能显著减少“结果漂移”。

## 3) Controls：控制不是装饰，是内部有效性的基线

控制策略应回答：如果结果变化了，你如何证明是方法导致，而非环境或数据泄漏？

常用控制措施：

1. 固定随机种子并报告种子列表；
2. 固定训练/测试划分方案；
3. 固定硬件和软件栈（容器镜像、内核、驱动）；
4. 对不同方法使用一致的超参数搜索预算；
5. 对有状态系统进行预热并去除冷启动偏差。

系统论文常见错误是：对新方法做了充分调优，对基线只跑默认配置。审稿时这通常会被定性为不公平比较。

## 4) Baselines：避免 strawman，构建“可被尊重的对照组”

推荐至少包含三层基线：

- **Simple baseline**：简单可解释（如规则法、最小实现）；
- **Strong baseline**：当前公开最强或被广泛认可的方法；
- **Operational baseline**：生产可用或行业常见方案（即使学术指标不是最强）。

系统安全例子：若你提出网络侧横向移动检测方案，不能只和一个过时签名规则比较。至少应加上近期图学习或时序检测方法，以及真实运维场景下的现有告警系统配置。基线选择原则不是“让自己赢”，而是“让结论经得起反方质疑”。

## 5) Metrics：指标必须对准主张

指标要按主张映射，而不是随手套通用表格：

- 主张“检测更准” → Precision/Recall/F1/AUC，并报告类别不平衡处理；
- 主张“开销更低” → CPU、内存、吞吐、P99延迟、尾部抖动；
- 主张“更稳健” → 在噪声、对抗扰动、分布漂移下的退化曲线；
- 主张“更可落地” → 部署时间、运维复杂度、参数敏感性。

对安全检测任务，仅报告AUC常常掩盖误报成本。应补充固定召回下的误报率，或单位时间告警量，以接近实际运营成本。

## 6) Dataset / Workload Selection：代表性优先于规模炫耀

数据集和工作负载应覆盖“你声称要解决的问题空间”。

建议做三层覆盖：

1. **Canonical datasets**：与文献可比；
2. **Realistic workloads**：接近真实生产分布；
3. **Stress workloads**：边界条件和压力场景。

示例（系统研究）：评估可重构交换机策略执行框架时，除标准流量回放外，加入突发流、长尾连接、策略频繁更新三类负载。这样才能证明“在最坏情况下仍可用”，而不是只在理想流量下有效。

## 7) Ablation Design：隔离贡献，而不是拼图式拆解

消融要验证“哪个机制带来哪个收益”。

最低要求：每次只移除一个关键组件，保持其余条件一致。常见设计包括：

- 移除特征选择模块，观察误报变化；
- 关闭在线更新，观察漂移场景下性能退化；
- 替换调度策略，观察尾延迟变化；
- 去掉安全强化策略，观察绕过攻击成功率变化。

错误做法是同时删多个模块，然后得出“模块都重要”。这类结论不可解释，也无法指导后续优化。

## 8) Statistical Test Selection：统计显著 ≠ 工程有意义

统计检验要根据数据分布、样本独立性和比较目标选择：

- 两组均值近似正态：t-test（必要时Welch）；
- 非正态或小样本：Mann–Whitney U；
- 多组比较：ANOVA或Kruskal–Wallis，配合事后检验；
- 比例差异：卡方或Fisher精确检验；
- 重复测量：配对检验或混合效应模型。

系统实验常见时间相关数据，简单独立同分布假设不成立。应考虑分块重采样或时间窗口聚合，避免夸大显著性。除p值外，建议同时报告效应量（Cohen’s d、Cliff’s delta）与置信区间。

## 9) Reproducibility Checklist：把“可复现”写成可执行条目

复现要求至少包括：

1. 代码版本（commit/tag）与依赖锁定；
2. 数据来源、预处理脚本和划分策略；
3. 环境信息（OS、内核、CPU、内存、NIC、驱动）；
4. 实验命令、随机种子、运行时参数；
5. 结果生成脚本（从原始日志到图表）；
6. 失败案例与已知限制说明。

安全领域常涉及敏感数据或攻击样本。若不能公开完整数据，应提供可替代公开子集、合成流程或可验证统计摘要，并清晰说明差异风险。

## 10) Systems/Security Concrete Example

假设研究命题是“基于XDP的网络侧策略执行框架可以在不显著增加延迟的前提下提高横向移动阻断率”。

- 假设：阻断率提升≥10%，P99延迟增加≤5%。
- 自变量：策略匹配机制（规则、哈希、分层索引）。
- 因变量：阻断率、误拦率、P99延迟、CPU占用。
- 控制变量：同一流量回放、同一硬件、同一策略集规模。
- 基线：iptables方案、传统交换ACL方案、近期eBPF方案。
- 指标：检测/阻断指标 + 性能指标 + 运维复杂度。
- 工作负载：办公网真实流量、攻防演练流量、突发流压力测试。
- 消融：去掉分层索引、去掉策略缓存、去掉动态更新。
- 统计：多次独立运行，报告置信区间和效应量。
- 复现：提供容器镜像、脚本、配置模板、日志样例。

这个结构能够直接支撑投稿时最关键的审稿问题：你是否真的提升了目标能力、比较是否公平、结果是否能重现、结论是否有边界。

## 11) 常见失败模式与纠偏

- **失败1：主张先行、实验补丁** → 先写假设-指标映射表再跑实验。
- **失败2：基线过弱** → 加入近期SOTA和工程可用方案。
- **失败3：只报平均值** → 必须补尾部指标与方差。
- **失败4：选择性报告** → 预注册主要指标，失败结果照样记录。
- **失败5：复现信息缺失** → 将复现清单当作交付件而非附录。

结论：实验设计的核心不是“跑出漂亮图”，而是让结论在反方审查下仍然成立。只要你能把每条贡献映射到假设、指标、基线和消融，再配上透明的复现条件，论文的说服力会明显上升。
